最近、GoのWebフレームワークであるEchoを使い始めたので、`echo.go` を読んでみました。

https://echo.labstack.com/

# 読んだバージョン

Echo v5.1.0

本記事で引用するコードは、`github.com/labstack/echo/v5@v5.1.0` の `echo.go` から抜粋したものです。

# 出発点となるサンプルコード

`echo.go` のパッケージコメントには、Echo の最小構成のサンプルが記載されています。

```go
package main

import (
	"log/slog"
	"net/http"

	"github.com/labstack/echo/v5"
	"github.com/labstack/echo/v5/middleware"
)

// Handler
func hello(c *echo.Context) error {
	return c.String(http.StatusOK, "Hello, World!")
}

func main() {
	// Echo instance
	e := echo.New()

	// Middleware
	e.Use(middleware.RequestLogger())
	e.Use(middleware.Recover())

	// Routes
	e.GET("/", hello)

	// Start server
	if err := e.Start(":8080"); err != nil {
		slog.Error("failed to start server", "error", err)
	}
}
```

このサンプルに登場する `echo.New()` / `e.Use(...)` / `e.GET(...)` / `e.Start(...)` の4つを、`echo.go` の実装と対応付けながら読み進めていきます。

# `echo.New()` — Echoインスタンスの生成

`Echo` はフレームワークのエントリーポイントとなる構造体です。`echo.New()` で最初に作るオブジェクトであり、`Router` / `Binder` / `Logger` / ミドルウェアスライス / Context プールといった部品をまとめています。

```go
// Echo is the top-level framework instance.
type Echo struct {
	serveHTTPFunc func(http.ResponseWriter, *http.Request)

	Binder Binder

	// Filesystem is the file system used for serving static files.
	Filesystem fs.FS

	Renderer         Renderer
	Validator        Validator
	JSONSerializer   JSONSerializer
	IPExtractor      IPExtractor
	OnAddRoute       func(route Route) error
	HTTPErrorHandler HTTPErrorHandler
	Logger           *slog.Logger

	contextPool sync.Pool

	router Router

	// premiddleware are middlewares that are called before routing is done
	premiddleware []MiddlewareFunc

	// middleware are middlewares that are called after routing is done and before handler is called
	middleware []MiddlewareFunc

	contextPathParamAllocSize atomic.Int32

	// formParseMaxMemory is passed to Context for multipart form parsing
	formParseMaxMemory int64
}
```

- **ロガー**: 標準 `log/slog` の `*slog.Logger` をそのまま持つ。
- **ファイルシステム**: 静的ファイル配信のために `fs.FS` を持つ。
- **ルーター**: `Router` はインターフェースで、`router` フィールドとして保持。
- **ミドルウェアのスライス**: ルーティング前の `premiddleware` とルーティング後の `middleware`。
- **コンテキストプール**: `contextPool sync.Pool` でリクエストごとの `*Context` を使い回す。
- **パスパラメータサイズ管理**: `contextPathParamAllocSize atomic.Int32` で、最大パスパラメータ数を追跡。
  - 後述の `add` メソッドでルート登録時にパラメータ数を見て、必要に応じて引き上げる仕組み。
- **`serveHTTPFunc`**: `ServeHTTP` の処理本体を差し替えられるように関数値で保持。

コメントに「**Goroutine safety: Do not mutate Echo instance fields after server has started.**」とあるとおり、 `echo.Start()` を呼んでサーバが起動した後は、フィールドを変更しない前提の設計になっています。

`echo.New()` はこの `Echo` 構造体のインスタンスを初期化して返します。

```go
// New creates an instance of Echo.
func New() *Echo {
	dir, _ := os.Getwd()
	logger := slog.New(slog.NewJSONHandler(os.Stdout, nil))
	e := &Echo{
		Logger:             logger,
		Filesystem:         NewDefaultFS(dir),
		Binder:             &DefaultBinder{},
		JSONSerializer:     &DefaultJSONSerializer{},
		formParseMaxMemory: defaultMemory,
	}

	e.serveHTTPFunc = e.serveHTTP
	e.router = NewRouter(RouterConfig{})
	e.HTTPErrorHandler = DefaultHTTPErrorHandler(false)
	e.contextPool.New = func() any {
		return newContext(nil, nil, e)
	}
	return e
}
```

- **ロガーは標準 `log/slog`** の JSON ハンドラをデフォルト採用。stdout に構造化ログを吐く。
- `Filesystem` として `NewDefaultFS(os.Getwd())` をデフォルトに。`fs.FS` ベースで静的ファイル配信を統一。
- `Router` は `NewRouter(RouterConfig{})` で生成。`Router` は **インターフェース**、`DefaultRouter` が具象型。
- `HTTPErrorHandler = DefaultHTTPErrorHandler(false)`。`DefaultHTTPErrorHandler` は **ファクトリ関数** で、引数 `exposeError bool` が内部エラーメッセージをクライアントに見せるかを決める。
- `contextPool` は `sync.Pool`。`*Context` を使い回してアロケーションを抑える。
- `serveHTTPFunc = e.serveHTTP` と関数値で持つ形になっており、`ServeHTTP` の振る舞いを差し替えやすい作りになっている。

設定を一気に渡したいときは `NewWithConfig(Config)` も用意されています。`Config` 構造体に `Logger` / `HTTPErrorHandler` / `Router` / `Filesystem` / `Binder` / `Validator` / `Renderer` / `JSONSerializer` / `IPExtractor` / `OnAddRoute` / `FormParseMaxMemory` を詰めて渡せば、`New()` の後にフィールドを書き換える形にならずに構築できます。

# `e.Use(...)` — ミドルウェアの登録

ハンドラとミドルウェアの型定義は次のとおりです。

```go
// HandlerFunc defines a function to serve HTTP requests.
type HandlerFunc func(c *Context) error

// MiddlewareFunc defines a function to process middleware.
type MiddlewareFunc func(next HandlerFunc) HandlerFunc
```

`*Context` を受け取る関数が `HandlerFunc`、それを包んで同じ型を返すのが `MiddlewareFunc`、というシンプルな構造です。

`Use` は、渡されたミドルウェアを `Echo.middleware` スライスの末尾に追加するだけの実装です。

```go
// Use adds middleware to the chain which is run after router has found matching route and before route/request handler method is executed.
func (e *Echo) Use(middleware ...MiddlewareFunc) {
	e.middleware = append(e.middleware, middleware...)
}
```

`Pre` という似たメソッドもあり、こちらはルーティング前に実行される `premiddleware` に追加されます。

```go
// Pre adds middleware to the chain which is run before router tries to find matching route.
// Meaning middleware is executed even for 404 (not found) cases.
func (e *Echo) Pre(middleware ...MiddlewareFunc) {
	e.premiddleware = append(e.premiddleware, middleware...)
}
```

premiddleware はルーティング前に走り、middleware はルーティング後に走るという違いがあります。premiddleware はルーティング前なので、404 ケースでも走る挙動になります。

実行時には, `serveHTTP` のなかで呼ばれる `applyMiddleware` 関数が `middleware` と `premiddleware` をそれぞれ適用してmiddlewareが適用された `HandlerFunc` を構築します。

```go
// serveHTTP implements `http.Handler` interface, which serves HTTP requests.
func (e *Echo) serveHTTP(w http.ResponseWriter, r *http.Request) {
	c := e.contextPool.Get().(*Context)
	defer e.contextPool.Put(c)

	c.Reset(r, w)
	var h HandlerFunc

	if e.premiddleware == nil {
		h = applyMiddleware(e.router.Route(c), e.middleware...)
	} else {
		h = func(cc *Context) error {
			h1 := applyMiddleware(e.router.Route(cc), e.middleware...)
			return h1(cc)
		}
		h = applyMiddleware(h, e.premiddleware...)
	}

	// Execute chain
	if err := h(c); err != nil {
		e.HTTPErrorHandler(c, err)
	}
}
```

```go
func applyMiddleware(h HandlerFunc, middleware ...MiddlewareFunc) HandlerFunc {
	for i := len(middleware) - 1; i >= 0; i-- {
		h = middleware[i](h)
	}
	return h
}
```

# `e.GET(...)` — ルート登録

`GET` をはじめとする各HTTPメソッドは、いずれも `Add` を呼び出す薄いラッパーで、戻り値は `RouteInfo` です。

```go
// GET registers a new GET route for a path with matching handler in the router
// with optional route-level middleware. Panics on error.
func (e *Echo) GET(path string, h HandlerFunc, m ...MiddlewareFunc) RouteInfo {
	return e.Add(http.MethodGet, path, h, m...)
}
```

`Add` は内部的に `Route` 構造体を組み立ててから `add` に渡し、`RouteInfo` と `error` のペアを受け取ります。`error` があれば **panic** するのが `Add` の挙動です。これにより、Route定義時にエラーがあればそもそもサーバが起動しないようになっているようです。

```go
// Add registers a new route for an HTTP method and path with matching handler
// in the router with optional route-level middleware.
func (e *Echo) Add(method, path string, handler HandlerFunc, middleware ...MiddlewareFunc) RouteInfo {
	ri, err := e.add(
		Route{
			Method:      method,
			Path:        path,
			Handler:     handler,
			Middlewares: middleware,
			Name:        "",
		},
	)
	if err != nil {
		panic(err) // this is how `v4` handles errors. `v5` has methods to have panic-free usage
	}
	return ri
}

func (e *Echo) add(route Route) (RouteInfo, error) {
	if e.OnAddRoute != nil {
		if err := e.OnAddRoute(route); err != nil {
			return RouteInfo{}, err
		}
	}

	ri, err := e.router.Add(route)
	if err != nil {
		return RouteInfo{}, err
	}

	paramsCount := int32(len(ri.Parameters)) // #nosec G115
	if paramsCount > e.contextPathParamAllocSize.Load() {
		e.contextPathParamAllocSize.Store(paramsCount)
	}
	return ri, nil
}
```

`add` の動きを整理すると次のとおりです。

- `OnAddRoute` フックが定義されていれば最初に呼ばれ、ここで **エラーを返せばルート登録を中断** できる。ルート検証やログ出力を差し込むポイント。
- ルート本体の登録は `e.router.Add(route)` に委譲。`Router` はインターフェースなので、差し替え可能。
- 登録時のパラメータ数 `len(ri.Parameters)` を見て、`contextPathParamAllocSize`（`atomic.Int32`）を **動的に引き上げる**。これが `Context` の `pathValues` のキャパシティとして使われる仕組みで、新しくルートを足しても古い Context が小さいスライスを持ち続けていれば次回プール補充時には新しい容量で作り直される。

# `e.Start(...)` — サーバ起動とリクエスト処理

`Start` は **`StartConfig` に委譲** する薄い実装です。

```go
// Start stars HTTP server on given address with Echo as a handler serving requests. The server can be shutdown by
// sending os.Interrupt signal with `ctrl+c`. Method returns only errors that are not http.ErrServerClosed.
func (e *Echo) Start(address string) error {
	sc := StartConfig{Address: address}
	ctx, cancel := signal.NotifyContext(stdContext.Background(), os.Interrupt, syscall.SIGTERM) // start shutdown process on ctrl+c
	defer cancel()
	return sc.Start(ctx, e)
}
```

ポイントは以下のとおりです。

- 内部で `signal.NotifyContext` を使って **Ctrl+C / SIGTERM を拾う**実装になっている。
- `http.ErrServerClosed` は返さない（正常終了扱い）。
- より細かい制御が必要なら `StartConfig` を自分で組んで `sc.Start(ctx, e)` を呼べばよい。`Address` / `HideBanner` / `HidePort` / `TLSConfig` / `GracefulTimeout` / `BeforeServeFunc` などがフィールドとして用意されている。

リクエスト処理のエントリポイントは `ServeHTTP` です。関数値 `serveHTTPFunc` 越しに呼び出す形になっていますが、実体はシンプルです。

```go
// ServeHTTP implements `http.Handler` interface, which serves HTTP requests.
func (e *Echo) ServeHTTP(w http.ResponseWriter, r *http.Request) {
	e.serveHTTPFunc(w, r)
}

// serveHTTP implements `http.Handler` interface, which serves HTTP requests.
func (e *Echo) serveHTTP(w http.ResponseWriter, r *http.Request) {
	c := e.contextPool.Get().(*Context)
	defer e.contextPool.Put(c)

	c.Reset(r, w)
	var h HandlerFunc

	if e.premiddleware == nil {
		h = applyMiddleware(e.router.Route(c), e.middleware...)
	} else {
		h = func(cc *Context) error {
			h1 := applyMiddleware(e.router.Route(cc), e.middleware...)
			return h1(cc)
		}
		h = applyMiddleware(h, e.premiddleware...)
	}

	// Execute chain
	if err := h(c); err != nil {
		e.HTTPErrorHandler(c, err)
	}
}
```

ここが Echo のリクエスト処理の中核です。読み解くと以下のようになります。

1. `contextPool` から `*Context` を取得し、`defer` で必ず返却。
2. `c.Reset(r, w)` で今回のリクエスト/レスポンスを流し込む。
3. `premiddleware` の有無で分岐する。
   - **なし**: `e.router.Route(c)` でルーティングしてハンドラを得る。`middleware` を適用したチェーンを作る。
   - **あり**: ルーティングと `middleware` 適用を内側の関数に包み、その外側に `premiddleware` を適用する。こうすることで、pre-middleware はルーティング前に実行でき、404 ケースでも走る挙動になる。
4. 構築したチェーン `h` を実行。エラーが返ったら `HTTPErrorHandler(c, err)` に委譲。引数順は **`(c *Context, err error)`**。

`Router.Route(c) HandlerFunc` は **「コンテキストを渡すとハンドラが返ってくる」** という形のインターフェース定義で、ルーターはルーティングの副作用として `c` にパスパラメータや `RouteInfo` を書き込みます。

# まとめ

- `*Echo` は `http.Handler` の実装で、`net/http` の `Server` や `StartConfig` から呼び出されるハンドラとして機能する。
- リクエスト処理のエントリは `ServeHTTP` → `serveHTTP`。ここでルーティング・ミドルウェアチェーン構築・実行・エラーハンドリングまで全て完結する。
- `contextPool` (`sync.Pool`) で `*Context` を使い回し、リクエストごとのアロケーションを抑制。`contextPathParamAllocSize` (`atomic.Int32`) でルート登録に応じてパスパラメータ用スライスのキャパシティを動的に更新する。
- ミドルウェアは `premiddleware`（ルーティング前、404 でも走る）と `middleware`（ルーティング後、ハンドラ直前）に分かれる。
- HTTPメソッド別の登録メソッドはすべて `Add` を経由し、戻り値は `RouteInfo`。panic-free に扱いたい場合は `AddRoute(Route)` を使う。
- サーバ設定は `Echo` 本体から切り離され、`StartConfig` に寄せられている。シンプルな起動だけなら `e.Start(":8080")` で Ctrl+C ハンドリングも含めて完結。
